# Feature Specification: AI Chat Mentions

**Feature Branch**: `003-ai-chat`
**Created**: 2025-12-28
**Status**: Draft
**Input**: User description: "我想要製作一個功能，當使用者在頻道發言時，當使用者mention了這個bot就會觸發ai聊天功能。ai會根據使用者的訊息發送回應，不需要保存過往訊息作為上下文，同時管理人員能夠通過.env文件配置base url和api，以及其他各項參數"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - AI 回應提及機器人訊息 (Priority: P1)

當使用者在 Discord 頻道中提及機器人時，機器人會使用 AI 服務生成並發送回應訊息到同一頻道。

**為何此優先級**: 這是核心功能，沒有此功能使用者無法與 AI 互動。

**獨立測試**: 可以透過在 Discord 頻道中提及機器人並驗證收到 AI 回應來獨立測試，提供即時 AI 對話能力。

**驗收場景**:

1. **Given** 使用者在頻道中傳送提及機器人的訊息，**When** 訊息被機器人接收，**Then** 機器人應在 5 秒內回傳 AI 生成的回應
2. **Given** 使用者提及機器人但訊息僅包含提及，**When** 機器人接收訊息，**Then** 機器人應回傳預設的 AI 問候語
3. **Given** 使用者在不同頻道提及機器人，**When** 訊息被接收，**Then** 機器人應在各自頻道回傳回應
4. **Given** 多位使用者同時提及機器人，**When** 訊息被接收，**Then** 每位使用者都應收到獨立的 AI 回應

---

### User Story 2 - 管理員配置 AI 服務參數 (Priority: P2)

管理員可以透過 .env 檔案配置 AI 服務的連線資訊與參數，包括 base URL、API 金鑰、模型名稱等。

**為何此優先級**: 配置功能允許管理員更換 AI 服務供應商或調整參數，但可先用預設值運行 P1 功能。

**獨立測試**: 可以透過修改 .env 檔案並重啟服務來驗證配置是否生效，測試不同 AI 服務的連線。

**驗收場景**:

1. **Given** .env 檔案包含有效的 AI 服務配置，**When** 服務啟動，**Then** 機器人應使用配置的 AI 服務
2. **Given** 管理員修改 .env 中的模型名稱，**When** 服務重啟後，**Then** 機器人應使用新模型生成回應
3. **Given** .env 檔案缺少必要的 API 金鑰，**When** 服務啟動，**Then** 服務應拒絕啟動並顯示明確的錯誤訊息
4. **Given** .env 檔案包含無效的 base URL，**When** 使用者提及機器人，**Then** 機器人應回傳錯誤提示訊息

---

### User Story 3 - 錯誤處理與降級 (Priority: P3)

當 AI 服務無法使用或發生錯誤時，系統應提供友善的錯誤回應給使用者。

**為何此優先級**: 錯誤處理提升使用者體驗，但不影響正常情況下的核心功能。

**獨立測試**: 可以透過模擬 AI 服務失敗（如無效的 API 金鑰、網路斷線）來驗證錯誤訊息。

**驗收場景**:

1. **Given** AI 服務回應 HTTP 401 未授權，**When** 使用者提及機器人，**Then** 機器人應回傳「AI 服務認證失敗，請聯絡管理員」
2. **Given** AI 服務回應 HTTP 429 速率限制，**When** 使用者提及機器人，**Then** 機器人應回傳「AI 服務暫時忙碌，請稍後再試」
3. **Given** AI 服務逾時或無法連線，**When** 使用者提及機器人，**Then** 機器人應回傳「AI 服務暫時無法使用」
4. **Given** AI 服務回應空內容，**When** 使用者提及機器人，**Then** 機器人應回傳「AI 沒有產生回應」

---

### Edge Cases

- 當使用者提及機器人的訊息超過 Discord 訊息長度限制時，系統應如何處理？
- 當 AI 生成的回應超過 Discord 單一訊息長度限制（2000 字元）時，應如何處理？
- 當機器人在未授權的頻道被提及時（缺少發言權限），系統應如何回應？
- 當 AI 服務回應時間超過 Discord 互動逾時（15 秒）時，應如何處理？
- 當 .env 配置的溫度（temperature）或最大 token 數超出有效範圍時，系統應如何處理？
- 當使用者在私訊（DM）中提及機器人時，是否應觸發 AI 回應？
- 當機器人被多次快速提及時，是否需要速率限制以防止 API 濫用？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須偵測使用者在頻道訊息中提及機器人的事件
- **FR-002**: 系統必項從訊息中提取使用者輸入的內容（排除機器人提及部分）
- **FR-003**: 系統必須將提取的訊息發送至 AI 服務 API 進行處理
- **FR-004**: 系統必須將 AI 生成的回應發送回原始頻道
- **FR-005**: 系統不得為每次請求儲存或使用過往訊息作為上下文
- **FR-006**: 系統必須支援透過 .env 檔案配置以下參數：
  - AI 服務 Base URL
  - API 金鑰
  - 模型名稱
  - 溫度（Temperature，控制回應隨機性）
  - 最大 Token 數（Max Tokens）
- **FR-007**: 當 AI 服務回應超過 Discord 訊息長度限制（2000 字元）時，系統必須將回應分割為多則訊息
- **FR-008**: 當 AI 服務呼叫失敗時，系統必須在頻道中顯示友善的錯誤訊息
- **FR-009**: 當 .env 檔案缺少必要配置時，系統必須在啟動時拒絕執行並顯示明確的錯誤訊息
- **FR-010**: 系統必須記錄所有 AI 服務呼叫的結果（成功或失敗）用於除錯

### Quality Requirements (Per Constitution)

- **QR-001**: 實作必須遵循測試驅動開發（先撰寫測試，再實作功能）
- **QR-002**: 程式碼必須達到最低 80% 測試覆蓋率（由 JaCoCo 測量）
- **QR-003**: 所有服務方法必須回傳 `Result<T, DomainError>` 進行錯誤處理
- **QR-004**: 新操作必須包含結構化日誌記錄與適當的日誌等級
- **QR-005**: 公開 API 必須包含 Javadoc 文件
- **QR-006**: 資料庫 schema 變更必須使用 Flyway 遷移（本功能不涉及資料庫變更）

### Key Entities

- **AI 聊天請求**：代表使用者的輸入訊息，包含頻道 ID、使用者 ID、訊息內容
- **AI 服務配置**：代表 AI 服務的連線與參數設定，包含 base URL、API 金鑰、模型名稱、溫度、最大 Token 數
- **AI 回應結果**：代表 AI 服務的回應，包含成功（回應內容）或失敗（錯誤類型與訊息）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者在提及機器人後，應在 5 秒內收到 AI 回應
- **SC-002**: 系統應能同時處理 100 個並行的提及請求而不降低服務品質
- **SC-003**: 95% 的 AI 服務呼叫應成功並產生可用的回應
- **SC-004**: 當 AI 服務無法使用時，使用者應在 3 秒內收到明確的錯誤提示
- **SC-005**: 管理員應能透過修改 .env 檔案並重啟服務來更換 AI 服務供應商
- **SC-006**: AI 回應超過 2000 字元時，應自動分割為多則訊息並保持內容完整性

## Assumptions

- 使用者的 Discord 訊息使用繁體中文
- AI 服務提供相容 OpenAI API 格式的端點
- 機器人在目標頻道擁有「傳送訊息」權限
- .env 配置變更需要重啟服務才能生效
- 不需要為不同使用者或頻道提供不同的 AI 配置
- 不需要過濾或審查 AI 生成的內容
- 不需要追蹤或限制每位使用者的請求次數

## Out of Scope

- 在私人訊息（DM）中使用 AI 聊天功能（僅支援公開頻道）
- 儲存對話歷史或提供上下文記憶
- 使用者自定義 AI 回應風格或人格設定
- 對話管理功能（如編輯、刪除 AI 回應）
- 與現有貨幣或遊戲代幣系統的整合
- 管理員指令即時更新配置（需重啟服務）
- 支援串流式 AI 回應（使用完整的請求-回應模式）
